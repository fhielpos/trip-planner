# Today View City Images — Design

## Goal

Give each day of the trip a distinct visual identity in the Today view by
showing a full-bleed illustrated background of the current city, instead of
the current plain accent-color gradient.

## Visual treatment

- Full-bleed background image behind the whole `.today-section`, in a flat
  vector-illustration style, but a full *scene* rather than a lone icon:
  a filled sky (gradient or color), a background layer (skyline/mountains/
  water), a midground with the landmark, and enough foreground detail that
  there's no empty/negative space. Vivid, saturated colors — richer than
  the original "muted" direction, which rendered too sparse. The palette
  should still lean toward the stay's existing `color` field (same color
  used for day cards, the legend, and the timeline) as an accent, not as a
  literal desaturation constraint.
- A CSS gradient scrim (transparent at the top, dark at the bottom/edges) is
  layered on top of the image in code, guaranteeing text legibility
  regardless of image content. No darkening is baked into the generated
  images themselves.
- **Desktop**: the image fills the section at its current compact height
  (the section stays a banner-height strip inside the existing
  `max-width: 1300px` layout — it does not become full-viewport).
- **Mobile**: the image fills the same near-full-screen `.today-section`
  that already exists (`calc(100dvh - 96px)`), so it reads as a genuine
  full-screen photo moment on phones.
- If a stay has no `image` value, the section falls back to the current
  plain accent-gradient background — no broken-image risk.

## Image inventory

One image per **unique city**, reused across repeat stays. The trip has 13
stays but only 11 unique cities (Buenos Aires and París are each visited
twice — once outbound, once on the return leg):

1. Buenos Aires
2. París
3. Loutraki
4. Viena
5. Salzburgo
6. Múnich
7. Lauterbrunnen
8. Zermatt
9. Chamonix
10. Ámsterdam
11. Bruselas

**Transit day:** the only day in the trip with no active stay is the final
checkout day (Oct 29, Buenos Aires return leg — `getActiveStay` treats
`check_out` as exclusive). No separate "in transit" illustration is needed;
that day resolves to the departure city's image (the stay whose
`check_out` equals today).

## Image generation spec

Handed off to a separate generating agent:

- **Format**: PNG, 1600×900 (16:9 landscape). (Originally specified as JPG;
  switched to PNG so a single extension covers both the photoreal images —
  see Status below — and the flat-vector placeholder generator, which
  produces cleaner flat-color output as lossless PNG than lossy JPEG.)
- **Style**: a complete, edge-to-edge scene — not an isolated landmark on
  empty/negative space. Every frame needs a filled sky, a background layer
  (skyline, mountains, water, hills — pick what fits the city), and
  foreground detail (streets, trees, people, boats, whatever suits the
  location) so the whole 1600×900 canvas reads as a place, not a logo.
  Vivid, saturated colors; golden-hour or twilight lighting reads well and
  gives good contrast for the text scrim. No text or watermarks baked in.
- **Composition**: subject centered in the middle third of the frame, so the
  same file survives a wide crop (desktop banner) and a tall portrait crop
  (mobile full-screen) via `background-size: cover; background-position: center`.
- **Filenames** (ASCII, lowercase, hyphenated):
  `buenos-aires.png`, `paris.png`, `loutraki.png`, `vienna.png`,
  `salzburg.png`, `munich.png`, `lauterbrunnen.png`, `zermatt.png`,
  `chamonix.png`, `amsterdam.png`, `brussels.png`.
- Files are placed in `data/images/` (created manually or by the generating
  agent) — already covered by the existing `./data:/app/data` docker-compose
  volume, no compose change needed.

### Status (2026-07-02)

`buenos-aires.png`, `paris.png`, and `loutraki.png` were produced externally
(a photorealistic/editorial style rather than the flat-vector direction
originally specified here) and are final. The remaining 8 cities are
temporary placeholder art (flat vector, procedurally composed per-city
silhouettes) generated by a now-removed local script, converted to PNG for
extension consistency, and are expected to be replaced the same way as the
first three — via an external image-generating agent, using the per-city
table below as the brief. The `image` field and rendering code don't care
which style a given file uses, so styles can mix freely while the rest are
swapped in.

### Correction: avoid procedural primitive-shape rendering

An early sample (`random.svg`, a scratch file in the repo root, not part of
this feature) showed the failure mode to avoid: a flat gray sky, flat
gray-purple ground, and one purple building
built from raw `rect`/`polygon`/`ellipse` primitives (a box, two stacked
triangles, a thin spire). With no color variation, texture, or
city-specific silhouette, it was unidentifiable — a generic spire could be
Vienna, Munich, Salzburg, or nowhere. If the generating tool composes
scenes procedurally from primitive shapes rather than actually rendering
requested imagery, it will keep producing this kind of unrecognizable
result no matter how the style prompt is worded. Each output must contain
enough real, distinguishing detail (specific silhouette, multiple color
values, layered depth) to identify the city on sight — that's a stronger
requirement than "not sparse."

### Per-city prompt

Reusable template, one call per city with the bracketed parts swapped:

> A vivid, richly colored flat vector illustration of **[landmark/scene]**
> in **[city]**. Full edge-to-edge scene filling the entire 1600×900 frame
> — no empty or negative space: a filled gradient sky, a background layer
> (skyline, mountains, or water as fits the city), and foreground detail
> (streets, trees, boats, people — whatever suits the location) for a sense
> of depth. Saturated, vivid color palette, golden-hour or twilight
> lighting. Landmark centered in the middle third of the frame. No text, no
> watermark, no logo.

### Ámsterdam — actual prompt used

The reusable template above was refined into this fuller, city-specific
prompt for `amsterdam.png`, generated with a reference image attached for
style consistency:

> Use the attached image as a style reference.
>
> Create a 1600×900 travel poster illustration.
>
> A vivid, richly colored flat vector illustration of Amsterdam.
>
> A picturesque canal lined with narrow gabled townhouses stretches through
> the center of the scene. Houseboats rest on the water and a cyclist
> crosses an elegant arched bridge in the foreground.
>
> Include trees, cafés, pedestrians, flowers, reflections in the canal, and
> warm city lighting.
>
> Fill the entire frame with foreground, midground, and background detail.
>
> Golden-hour lighting, saturated colors, rich gradients, highly detailed
> vector artwork, modern travel-poster style.
>
> The canal and bridge are centered in the middle third of the frame.
>
> No text, no watermark, no logo.

## Data model changes

- `data/accommodations.json`: add an `"image": "<filename>.png"` field to
  each of the 13 entries, same pattern as the existing `color`/`url`
  fields. The two Buenos Aires entries and the two París entries point to
  the same filename.
- No schema/endpoint change needed for the field to reach the client:
  `GET /api/accommodations` already returns raw entries, and
  `public/js/app.js` already fetches and merges `tripData.accommodations`
  client-side, so `image` flows through automatically.
- The `POST /api/accommodations` field whitelist is intentionally left
  unchanged — images are added by editing the JSON file directly (matching
  how this data is currently managed), not through the app's add/edit UI.

## Server changes

- `server.js`: add one static route to expose the new images directory —
  `app.use('/images', express.static(path.join(__dirname, 'data', 'images')))`.
  This exposes only `data/images/`, not the rest of `data/` (which holds
  JSON files not meant to be served).

## Frontend changes

- `public/js/today.js` — `renderToday()`:
  - Resolve the relevant stay: the active stay if one exists, otherwise the
    stay whose `check_out` equals today (transit/checkout day).
  - Set the resolved stay's image as a CSS custom property, e.g.
    `section.style.setProperty('--today-image', stay?.image ? \`url(/images/${stay.image})\` : '')`.
  - No image → property is empty, CSS falls back to the plain gradient.
- `public/css/styles.css`:
  - `.today-section` gains an additional background layer using
    `var(--today-image)` plus the scrim gradient, composited with the
    existing `var(--today-bg, var(--accent-dim))` gradient.
  - The existing mobile media query (`@media (max-width: 640px)`) keeps its
    current full-height flex layout; the image simply renders behind it.

## Non-goals

- No lazy-loading, responsive `srcset`, or CDN optimization — 11 files at
  ~200KB each is negligible.
- No in-app upload/edit flow for the `image` field — it's set once by
  editing `accommodations.json`, consistent with how `color` is managed
  today.
- No second image size/orientation per city — a single centered 1600×900
  file is cropped via CSS on both breakpoints.
